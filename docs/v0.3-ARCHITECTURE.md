# v0.3 Architecture — Editorial Layout Engine

The full design rationale lives in [the spec](superpowers/specs/2026-05-10-openmagazine-v0.3-editorial-engine-design.md). This page is the user-facing summary.

## What v0.3 adds

| Capability | v0.1 / v0.2 | v0.3 |
|---|---|---|
| Layout style | Full-bleed photos with typography painted into the image | Editorial spreads with multi-image regions + real PDF text |
| PDF engine | ReportLab (imperative) | WeasyPrint (HTML/CSS Paged Media, declarative) |
| Layout schema | yaml metadata only | yaml + Jinja2 HTML templates |
| Image gen | One 4K image per page | 21 images per issue, multi-aspect (cover hero, scenes, portraits, environment, detail, back coda) |
| Typography | Generated INTO the photo | Real PDF text with embedded fonts, drop caps, accent rules, page numbers |

Both modes coexist. Legacy `plain-4` / `plain-16` (schema v1) layouts keep working via ReportLab; the new `editorial-16page` (schema v2) routes through WeasyPrint. `tools/pdf/pdf_selector.py` picks the engine based on `layout.schema_version`.

## How to choose

- **Use editorial v0.3** when you want a magazine-as-a-book feel: feature articles, real text, multiple images per spread, typography, drop caps.
- **Use plain v0.1/v0.2** when you want a photo book: every page is one large photograph; typography is part of the image.

## Pipeline differences

`editorial-16page` introduces a NEW stage `articulate` between proposal and storyboard:

```
v0.2 smoke-test-4page:   research → proposal → storyboard → upscale → compose → publish
v0.3 editorial-16page:   research → proposal → articulate → storyboard → upscale → compose → publish
                                              ↑
                              article copy drafted here, user reviews + edits
                              before any image gen spend
```

The articulate stage produces `library/articles/<slug>.yaml` containing all editorial copy + per-spread `image_slot_overrides` (scene descriptions per image slot). Storyboard and upscale read these so each of the 21 slots gets a slot-specific scene.

## Cost (editorial-16page)

- **21 image slots** × $0.24 (Vertex Gemini 3 Pro Image) = $5.04
- 1 codex storyboard inference ≈ $0.04
- **Total ≈ $5.08 per issue**, $5.50 default budget cap with 8% headroom
- Wall time ≈ 15 min (storyboard 4 min, upscale 9 min concurrent at parallelism 3, compose < 1 min)

## Verifying v0.3 locally

Without spending Vertex budget — the integration test renders a 16-page PDF using placeholder PNGs:

```bash
cd ~/github/openMagazine && source .venv/bin/activate
python -m pytest tests/integration/test_render_dry_run.py -v
```

This exercises every spread component (rendered to disk), every typography rule (drop cap, accent, page numbers), and the WeasyPrint pipeline end-to-end. If this passes, the only remaining live-run risk is image-gen quality.

## Known limits (v0.3.0)

- One layout (`editorial-16page`); 4-page and 32-page editorial variants are v0.3.1+.
- Two brand presets (`editorial-classic`, `humanist-warm`); 3 more in v0.3.1+.
- Six spread types; chapter-opener / index / advertorial / two-column-feature deferred.
- RGB only; CMYK output deferred.
- One language per issue (no bilingual side-by-side spreads yet).

See [`docs/ROADMAP.md`](ROADMAP.md) for the v0.3.1 / v0.4 path.

## v0.3.1: Regions data layer

Each editorial spread is composed of **regions** declared in
`library/layouts/_components/<type>.regions.yaml`. The same data is read
by three consumers:

- **WeasyPrint render** — positions absolute divs from `rect_norm`
- **Image gen prompts** — model is told sibling regions to keep calm
- **`article_validate`** — every `role: text` region must have its
  `text_field` populated

See [regions-reference.md](regions-reference.md) and
[component-registry-reference.md](component-registry-reference.md) for
catalogues. The design rationale lives in
[`docs/superpowers/specs/2026-05-11-regions-as-shared-contract-design.md`](superpowers/specs/2026-05-11-regions-as-shared-contract-design.md).

All 7 spread types (cover / toc / feature-spread / pull-quote /
portrait-wall / colophon / back-cover) ship as regions-driven in v0.3.1.

## v0.3.2: Design contracts as shared data layer

Two new yaml layers parallel to v0.3.1's regions:

- `library/profiles/<name>.yaml` — publication type (closed registry,
  ships `consumer-retail` in v0.3.2)
- `library/design-systems/<slug>.yaml` — per-issue resolved decisions
  (typography fallback chain, text-safe contracts, brand authenticity
  gates, output targets)

The same upstream pipeline (stages 1-5) now feeds **multiple realizers**
at the compose stage, routed by `design_system.output_targets`:

| target | realizer | output |
|---|---|---|
| `a4-magazine` | WeasyprintCompose | `magazine.pdf` |
| `photobook-plain` | ReportlabCompose | `magazine.pdf` (v1 path) |
| `deck-pptx` | PresentationsAdapter | `deck/<slug>.pptx` |

`tools/output/output_selector.py` replaces v0.3.1's `pdf_selector.py`
(kept as backward-compat shim).

See [regions-as-shared-contract](superpowers/specs/2026-05-11-regions-as-shared-contract-design.md)
and [design-contracts-multi-realizer](superpowers/specs/2026-05-13-design-contracts-multi-realizer-design.md)
for the two consecutive applications of the same architectural pattern:
**lifting decisions out of actor internals into shared yaml data**.

See also: [design-system-reference.md](design-system-reference.md),
[profiles-reference.md](profiles-reference.md).
